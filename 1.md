
Правильная последовательность такая:

Шаг 1 — зафиксировать доменную модель.
Пока не надо писать роуты. Сначала нужно письменно утвердить, что у вас есть Project, Node, Edge, Lane, RoadType, Connection, Intersection, Movement, Phase. Это вытекает из ТЗ: сеть — это граф, участок — однонаправленная дорога с полосами, пересечение настраивается поверх узла, а хранение должно быть совместимо с .nod.xml, .edg.xml, .typ.xml, .con.xml.

Шаг 2 — утвердить внутренний JSON-контракт.
Даже если финальная совместимость нужна с XML, внутри сервера и между командами удобнее жить в JSON DTO. Потом отдельный слой делает export/import XML. Это позволит backend и команде логики работать параллельно без постоянной боли с XML. Требование к XML остаётся, но не должно определять все внутренние API-вызовы.

Шаг 3 — поднять каркас backend.
Сразу нужны:
	•	healthcheck
	•	project CRUD
	•	node CRUD
	•	edge CRUD
	•	lane CRUD внутри edge
	•	connection CRUD
	•	export XML

Без этого логическая команда будет считать “в воздухе”.

Шаг 4 — сделать editor operations.
Не только CRUD, а операции уровня конструктора:
	•	создать узел по клику
	•	соединить 2 узла участком
	•	создать двустороннюю дорогу как 2 directed edge
	•	изменить shape участка
	•	превратить узел в перекрёсток
	•	получить snapshot проекта

Это напрямую соответствует режиму “Конструктор” из ТЗ.

Шаг 5 — подключить пересечения и регулирование.
После того как сеть живая, добавляете:
	•	intersection config
	•	movements
	•	priorities
	•	crosswalks
	•	phases

Потому что в ТЗ пересечение зависит от подключённых участков, направлений, приоритетов и регулирования.

2. Подход к разработке на FastAPI + Postgres

Под хакатон я бы рекомендовал модульный монолит, а не onion-архитектуру в полном объёме.

Почему:
	•	быстрее старт
	•	проще онбординг для команды
	•	удобно тестировать
	•	не разваливается на 20 слоёв

Для FastAPI актуально использовать:
	•	lifespan вместо старых startup/shutdown handlers — это сейчас рекомендуемый способ инициализации приложения; FastAPI прямо пишет, что при наличии lifespan старые startup/shutdown лучше не смешивать.  ￼
	•	Dependency Injection через Depends для сессии БД, сервисов и текущего пользователя — это базовый рекомендованный механизм FastAPI.  ￼
	•	pydantic-settings / Pydantic v2 для конфигурации и DTO; Pydantic v2 — текущая основа экосистемы, а FastAPI в свежих версиях продолжает упрощать и чистить совместимость именно под v2.  ￼
	•	SQLAlchemy 2 style, в том числе async_sessionmaker, если вы идёте в async-стек; это официальный современный путь для асинхронной ORM-работы.  ￼

Но для хакатона важный practical point:
не надо брать async только потому, что это “модно”. Если вся команда не уверенно живёт в async SQLAlchemy, то можно сделать:
	•	async FastAPI роуты
	•	но простую и понятную работу с БД без лишних абстракций

Главная цель хакатона — скорость и предсказуемость, а не архитектурная идеальность. FastAPI даёт production-ready API и при этом остаётся быстрым в разработке.  ￼

6. Какие инварианты зашить сразу

Это особенно важно для хакатона, потому что потом они спасают от мусора в данных.

Нужно сразу проверять:
	•	edge нельзя создать без from_node_id и to_node_id
	•	edge не может ссылаться на node из другого проекта
	•	двусторонняя дорога — это два edge
	•	у edge должна быть минимум одна lane
	•	lane_index внутри edge должен быть уникален и идти от 0..N-1
	•	connection можно создать только если обе полосы реально существуют
	•	node нельзя удалить, если на него ссылаются edge
	•	edge нельзя удалить, если на него ссылаются connection или movement
	•	phase не может ссылаться на несуществующий movement

Эти правила напрямую поддерживают графовую модель сети и требования по редактированию и хранению сущностей.

Там уже:
	•	проверки
	•	graph operations
	•	generation logic
	•	snapshot
	•	XML export

Слой 5 — Router

Роут максимально тонкий:
	•	принял DTO
	•	вызвал сервис
	•	вернул DTO

Это сейчас наиболее живой и практичный паттерн для FastAPI-проектов: активно использовать типизацию, DI и чётко разделять transport layer и business logic. FastAPI опирается на стандартные type hints и DI, Pydantic v2 даёт быстрые модели валидации, а SQLAlchemy 2 сохраняет современный typed-style работы с ORM.  ￼

12. Мой итоговый совет

Под ваш кейс лучший подход такой:

FastAPI как тонкий API + сервисный слой.
Postgres как источник истины для дорожного графа.
JSON как внутренний рабочий контракт.
XML как обязательный слой совместимости с SUMO.
Файловая структура — модульный монолит, а не “всё в одном” и не “чрезмерный enterprise”.

Это будет достаточно современно, достаточно быстро и очень хорошо подходит именно под хакатон.
