# Анализ фронтенда: функционал и техническая реализация

## 1. Общая архитектура

- Стек: `React 19 + TypeScript + Vite + Tailwind CSS v4 + Leaflet/react-leaflet`.
- Точка входа: `src/main.tsx` -> `App.tsx` -> `MapEditor`.
- Весь прикладной функционал фронта сосредоточен в:
  - `src/prototype/components/MapEditor.tsx`
  - `src/prototype/components/Sidebar.tsx`
  - `src/prototype/utils/osmParser.ts`
  - `src/prototype/utils/graphValidation.ts`
  - `src/prototype/types.ts`

## 2. Пользовательский функционал

### 2.1 Режимы редактора

В редакторе доступны режимы:

- `SELECT` - выбор и перемещение узлов/точек рёбер.
- `ADD_NODE` - добавление обычного узла полосы.
- `ADD_TRAFFIC_LIGHT` - добавление светофора.
- `ADD_CROSSING` - добавление пешеходного перехода.
- `ADD_BUS_STOP` - добавление остановки.
- `ADD_SPEED_LIMIT` - добавление знака ограничения скорости (с вводом значения).
- `ADD_EDGE` - создание ребра (полосы) между двумя узлами.
- `DELETE` - удаление узлов, рёбер и промежуточных точек.

### 2.2 Работа с узлами

- Добавление узлов по клику на карте (тип зависит от активного режима).
- Перетаскивание узлов мышью в режиме выбора.
- Выбор узла кликом.
- Удаление узла в режиме удаления:
  - удаляется сам узел;
  - каскадно удаляются все связанные рёбра.

### 2.3 Работа с рёбрами (полосами)

- Создание ребра последовательным выбором 2 узлов в режиме `ADD_EDGE`.
- Выбор ребра кликом.
- Удаление ребра в режиме `DELETE`.
- Переключение признака односторонности (`isOneWay`) для выбранного ребра в сайдбаре.
- Добавление промежуточной точки ребра правым кликом по выделенному ребру.
- Перетаскивание промежуточных точек.
- Удаление промежуточной точки кликом по ней в режиме удаления.

### 2.4 Импорт OSM

- Есть загрузка `.osm` файла через сайдбар.
- После импорта:
  - XML парсится в внутренний формат `NetworkState`;
  - карта автоматически центрируется по среднему центру импортированных узлов.
- Импорт строит lane-level сеть:
  - учитывает `highway`, `oneway`, `lanes`, `lanes:forward`, `lanes:backward`, `name`, прочие теги;
  - исключает служебные/пешеходные типы дорог (`footway`, `path`, `cycleway`, и др.);
  - создаёт отдельные lane-узлы с геометрическим оффсетом по нормали;
  - создаёт направленные lane-рёбра;
  - формирует соединения на пересечениях;
  - добавляет специальные узлы (светофоры/переходы/остановки) из OSM-тегов.

### 2.5 Проверка сети

- Кнопка "Проверить сеть" запускает валидацию.
- Реализованы проверки:
  - связность графа (BFS от первого узла);
  - изолированные узлы (степень 0).
- Результат показывается в сайдбаре как предупреждения/успех.

### 2.6 Ограничения скорости

- В режиме `ADD_SPEED_LIMIT` по клику на карту открывается модальное окно для ввода скорости.
- Проверка ввода: только положительное целое число, затем ограничение в диапазоне `5..200`.
- На карте отображается круглый знак со значением.
- Для каждого lane-ребра вычисляется эффективное ограничение:
  - по умолчанию `60`;
  - знак применяется к ближайшим правым полосам в радиусе `15 м`;
  - затем ограничение распространяется по графу вперёд (BFS), пока не достигнут узел-пересечение;
  - при конфликте нескольких знаков приоритет у меньшей глубины обхода, затем у более близкого стартового расстояния.

### 2.7 Визуальные подсказки и семантика карты

- Разные иконки для типов узлов: обычный, светофор, переход, остановка, ограничение скорости.
- Для обычных узлов рисуются стрелки направления движения (вычисляются из геометрии lane-рёбер).
- Для пешеходных переходов рисуется "зебра":
  - ориентация определяется по ближайшей геометрии полосы;
  - число штрихов адаптируется к числу соседних полос;
  - масштаб зависит от текущего зума карты.
- Рёбра рядом с переходом/остановкой подсвечиваются другим цветом.
- Tooltip на ребре показывает:
  - имя;
  - ограничение скорости;
  - предупреждение о близости перехода/остановки;
  - OSM-теги ребра.

### 2.8 История изменений

- Поддерживаются `Undo/Redo`.
- Реализовано snapshot-подходом:
  - каждое действие пушит полное состояние `NetworkState` в массив истории;
  - есть текущий индекс в истории;
  - при новом действии после `Undo` "ветка будущего" отбрасывается.

## 3. Техническая реализация

### 3.1 Модель данных

- `NetworkState`:
  - `nodes: Record<string, Node>`
  - `edges: Record<string, Edge>`
- `Node`: координаты, тип, имя, `speedLimit`.
- `Edge`: `sourceId/targetId`, промежуточные точки, односторонность, lane-метаданные (`laneIndex`, `isForward`), OSM-теги.

### 3.2 Компонентная структура

- `MapEditor`:
  - хранит основной state, выбранные сущности, режим редактора, зум, результаты валидации, состояние модалки ограничения скорости;
  - реализует все map handlers и вычисления.
- `Sidebar`:
  - режимы, undo/redo, импорт OSM, запуск валидации;
  - редактирование свойства `isOneWay` выбранного ребра;
  - вывод результатов валидации.
- `MapCenterer`:
  - изолированная логика центровки карты после импорта.

### 3.3 Рендер карты

- Карта на `react-leaflet`:
  - `MapContainer`, `TileLayer`, `Marker`, `Polyline`, `Tooltip`, `useMapEvents`, `useMap`.
- Включены `maxZoom=22`, `maxNativeZoom=19`.
- Для выбора ребра используется невидимая "толстая" полилиния поверх основной, чтобы увеличить hit-area.

### 3.4 Вычислительные алгоритмы

- `useMemo` для дорогих вычислений:
  - `nodeDirectionAngles` - угол стрелки направления по геометрии ребра;
  - `edgeSpeedLimits` - назначение/протяжка ограничений скорости через BFS;
  - `crossingZebraStripes` - генерация геометрии штрихов перехода.
- Геометрические преобразования `lat/lng <-> local XY` используются для метрических расчётов расстояний/сторон.

### 3.5 Импорт OSM (детали)

- Парсинг XML через `DOMParser`.
- Детектирование специальных узлов по тегам:
  - `highway=traffic_signals`
  - `highway=crossing`
  - `highway=bus_stop` и `public_transport=platform`
- Для way:
  - сбор тегов в `tags`;
  - расчёт направления полос;
  - оффсет lane-геометрии с шагом `~3.5 м`.
- Между lane-узлами одного исходного OSM-узла на пересечениях создаются соединительные рёбра.

## 4. Технические ограничения и риски

- История хранит полные снепшоты, что может расти по памяти на крупных графах.
- В проекте есть `useNetworkState` (готовый хук истории), но фактически в `MapEditor` используется дублирующая inline-реализация.
- Часть русских строк в исходниках сохранена в "битой" кодировке (mojibake), это влияет на читаемость UI-текста.
- Нет интеграции фронта с backend API: редактор работает локально в памяти, без сохранения проекта на сервер.

## 5. Вывод

Текущий фронт - это полнофункциональный интерактивный прототип lane-level редактора дорожной сети на карте с импортом OSM, ручным редактированием топологии, базовой валидацией и расширенной визуальной семантикой (переходы, остановки, ограничения скорости, стрелки направления).
